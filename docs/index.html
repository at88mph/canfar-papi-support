CANFAR Science Platform & the SKAO Permissions API

Configuring the CANFAR Science Platform to integrate with the SKAO Permissions API to enable SKAO-style permission checks.

Dustin Jenkins (Team Coral)

Current system

Today, access is decided inside the Skaha webservice using Group-based access control.

1

User calls skaha

HTTP request with bearer token.

→
2

Extract groups

Call to IVOA GMS to augment the user's set of credentials.

→
3

Match configured rule

Compare user's memberships with configured IVOA Group URI.

→
Allowed — request proceeds.
Denied — no match → e.g. 403.

Current system (continued)

User storage is managed by the Cavern webservice. Skaha calls Cavern to run the administrative operation of creating a new user allocation; trust is established with a shared API key.

1

Skaha calls Cavern

Skaha is configured with an API key. When a user requests a Session, Skaha calls Cavern to verify the presence of a Home directory for the user. One is created using the Authorization: admin-api-key header, if none exists.

→
2

Cavern verifies the key

Cavern compares the presented key to a set of trusted values held in its own configuration before accepting the administrative action.

→
Allowed — key matches a trusted value; Cavern creates the user allocation.
Denied — key not trusted → allocation is not created (e.g. 401 / 403).

Changes

Building on the current system slide: group-only checks inside Skaha give way to the centrally managed Permissions API — token exchange, then a dedicated authorization call for the user behind the request.

  • 1

    User calls Skaha

    The client presents a Bearer token on the Skaha request.

  • 2

    Skaha requests token exchange

    Skaha calls the Permissions API to request exchanging the user’s token for an API-facing credential.

  • 3

    If approved: exchange

    When the API approves, Skaha completes the token exchange for that user.

  • 4

    Authorization at the API

    Skaha uses the exchanged token to ask the Permissions API whether this user has access for the operation at hand.

  • Allowed — API grants authorization; Skaha continues the user request.
    Denied — exchange or authorization fails → e.g. 401 / 403.
  • 1

    User calls Cavern to request a new allocation

    The client presents a Bearer token on the Cavern request.

  • 2

    Cavern requests token exchange

    Cavern calls the Permissions API to request exchanging the user’s token for an API-facing credential.

  • 3

    If approved: exchange

    When the API approves, Cavern completes the token exchange for that user.

  • 4

    Authorization at the API

    Cavern uses the exchanged token to ask the Permissions API whether this user has permission to create a new allocation.

  • Allowed — API grants authorization; Cavern creates the new allocation.
    Denied — exchange or authorization fails → e.g. 401 / 403.

What users should notice

  • The portal and primary workflows can look unchanged for authorized users; enforcement moves behind the scenes to the Permissions API.
  • Same happy path: sessions and storage allocation succeed when centralized policy allows—still driven by the user's identity and Bearer token on Skaha and Cavern.
  • Operational clarity: fewer bespoke paths (group URI only inside Skaha; admin API key to Cavern where retired)—auditing lines up with Permissions API routes configured in Helm.
  • Failures may surface as familiar HTTP errors (e.g. 403); semantics should reflect centralized policy rather than only local checks—details belong in ops/runbooks.

Skaha Modifications

Web Service

  • Accept configuration for the Permissions API (OpenCADC cadc-permissions-client)
  • Default to existing Group check configuration
  • Remove Admin API key request to Cavern

Helm Chart

  • Add new configuration for the Permissions API
  • Remove Admin API key request to Cavern

Skaha Helm Chart Modifications

Previously used usersGroup to configure the Group check.

values.yaml 
            
              deployment:
                skaha:
                  sessions:
                    authorization:
                      permissionsAPI:
                        enabled: true
                        baseURL: "https://permissions.srcnet.skao.int/api"
                        authAPIBaseURL: "https://authn.srcnet.skao.int/api"
                        type: "route"
                        route: "/skaha"
                        name: "canfar-api"
                        version: "1"
                        method: "GET"
                      group:
                        enabled: false
                        uri: "ivo://skao.int/gms?roles/prod/user"

Operational considerations

Central policy is clearer for security and ops.

  • Risk: each protected operation can incur about three extra network calls compared with legacy checks—in practice, latency, timeouts, and failure modes tie services to Permissions API availability and behaviour.
  • Mitigate: explore caching with TTLs (Time-To-Live), potentially keyed by resource and identity.
  • Who to contact for access: entitlement changes live in one place—point people to how they should request permission (e.g. Send request to access-control@srcnet.skao.int).
